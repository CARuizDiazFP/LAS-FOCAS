# PR Diario — 2026-04-30

## Resumen de Cambios

Refactoring completo de la capa de acceso a datos de la API FastAPI para eliminar bloqueos en el event loop. Se adoptó `asyncpg` como driver asíncrono y `SQLAlchemy 2.0` async (`create_async_engine`, `AsyncSession`, `async_sessionmaker`). Todos los route handlers de la API fueron convertidos a `async def` con `Depends(get_async_db)` o `asyncio.to_thread()` según la complejidad del servicio subyacente. El worker de Slack Baneo Notifier fue migrado de `BlockingScheduler` a `AsyncIOScheduler`.

---

## Contexto y Alcance

- **Módulos afectados**: `db/`, `api/app/`, `modules/slack_baneo_notifier/`, `docs/api.md`
- **Objetivo**: Hacer la API completamente no bloqueante usando `asyncpg==0.29.0` + SQLAlchemy 2.0 async, sin romper workers sincrónicos ni Alembic.
- **Supuesto clave**: `core/services/infra_service.py` y `core/services/protection_service.py` son complejos y conservan sesiones síncronas (`Session`); se accede a ellos desde la API vía `asyncio.to_thread()`.
- **Driver `psycopg[binary]`**: se mantiene para el engine síncrono (workers, Alembic). No se elimina.

- Patrón **dual-engine**:

## Decisiones Técnicas

1. **Opción B — auto-swap de URL**: `db/session.py` deriva la URL asyncpg reemplazando el prefijo del driver. No requiere `ASYNC_DATABASE_URL` separada.
2. **No se convierte `infra_service.py` ni `protection_service.py`**: contienen cientos de líneas de ORM con lazy-loading complejo; el costo de conversión supera el beneficio para esta iteración. Se acceden vía `asyncio.to_thread`.
3. **`expire_on_commit=False`** en `async_sessionmaker`: evita `MissingGreenlet` al acceder a atributos fuera del contexto de sesión.
