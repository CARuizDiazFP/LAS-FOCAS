# Descripción: Registro de cambios para la versión de Enero 2026

## 1. Resumen de cambios

- **Notificaciones de Baneo:** El modal de correo ahora carga los datos del incidente (asunto/cuerpo persistidos y cantidad real de cámaras) desde el backend y muestra el conteo correcto en la plantilla.
- **Corrección UX:** Se expone correctamente `openEmailModal` desde `panel.js` para evitar el error de referencia al abrir el editor de correo desde el botón "Dar Aviso".
- **Robustez del modal:** `openEmailModal` admite fallback con datos del listado activo si el endpoint de detalle responde 404/Not Found, evitando bloquear al usuario y mostrando la plantilla con valores por defecto.
- **Persistencia en descarga EML:** El frontend ahora envía `subject` y `html_body` como `form-data` al endpoint `/api/infra/notify/download-eml`, asegurando que se guarden en DB antes de generar el archivo y se use el conteo real de cámaras (`cantidad_camaras`) en la plantilla.
- **Fix 422 envío directo:** El botón “Enviar Correo” ahora envía el payload que espera `/api/infra/notify/send` (`to[]`, `subject`, `body`, `incidente_ids`, `include_xls/txt`), evitando errores 422 y manteniendo el cuerpo con la cantidad de cámaras real.
- **Compatibilidad de conteo:** `openEmailModal` ahora toma el conteo desde múltiples claves (`cantidad_camaras`, `camaras_count`, `total_camaras` o largo de `camaras_afectadas`) para soportar respuestas previas del endpoint y garantizar que el texto muestre N cámaras en lugar de “?”.
- **Fix validación 422:** El endpoint `/api/infra/notify/download-eml` tipa `recipients/subject/html_body` como opcionales (`Optional[str] = Form(None)`), evitando errores de validación cuando faltan campos al descargar el EML y persistiendo los cambios antes de generar el archivo.
- **Frontend modal correo:** El FormData de descarga usa claves snake_case (`incident_id`, `html_body`, `subject`, `recipients`) y asegura strings válidas; el modal muestra “Cargando datos...” mientras obtiene el detalle y rellena `{{cantidad}}` con el valor devuelto por el backend.
- **Servicio de email:** `EmailService.generate_ban_eml` usa la lista explícita `camaras_afectadas` que recibe del endpoint (sin acceder a atributos inexistentes) para construir el Excel y cuerpo del EML, alineado con la nueva API.
- **Refactorización de Ingesta de Trackings FO:** Se ha mejorado el sistema para incluir Puntos Terminales (puntas A/B), alias para pelos/hilos y una detección inteligente de conflictos (`POTENTIAL_UPGRADE`, `NEW_STRAND`).
- **Actualización de UI en Panel de Infraestructura:** La interfaz ahora gestiona los nuevos escenarios de conflicto y visualiza metadatos enriquecidos en las tarjetas de cámara (alias, cantidad de pelos, tránsitos).
- **Corrección Crítica en Informe SLA:** Se ha solucionado un bug mayor en el cálculo de horas, asegurando que el informe utilice la columna "Horas Netas Reclamo" (columna U) del Excel fuente, restaurando la precisión de las métricas.
- **Mejora de UX en Notificaciones:** Se simplificó el modal "Dar Aviso" para la composición de correos, haciéndolo más intuitivo y funcional.

## 2. Contexto y alcance

- **Módulos afectados:** `core/parsers`, `core/services`, `db/models`, `api/routes`, `web` (frontend y backend).
- **Objetivo:** Aumentar la inteligencia y robustez del sistema de gestión de infraestructura de fibra óptica, y corregir un bug crítico que afectaba la veracidad de los informes de SLA.

## 3. Cambios realizados

- **Esquema DB:**
  - Migración `20260127_01_incidente_baneo_email_fields.py` agrega `email_subject` y `email_body` a `app.incidentes_baneo`.
  - Nueva tabla `PuntoTerminal` y migración Alembic `20260113_01_puntos_terminales.py`.
  - Se añadieron los campos `alias_ids` y `cantidad_pelos` al modelo `RutaServicio`.

- **Backend:**
  - `api/api_app/routes/infra.py`: Nuevo endpoint `GET /api/infra/ban/{incidente_id}` devuelve asunto, cuerpo y cantidad de cámaras; `/api/infra/notify/download-eml` ahora persiste asunto/cuerpo antes de generar el EML.
  - `core/parsers/tracking_parser.py`: Ahora soporta puntas A/B, alias, tránsitos y cantidad de pelos.
  - `core/services/infra_service.py`: Nueva lógica para detectar y resolver conflictos de `POTENTIAL_UPGRADE` y `NEW_STRAND`.
  - `api/api_app/routes/infra.py`: Endpoints `analyze` y `resolve` actualizados para los nuevos escenarios.
  - `core/sla/legacy_report.py`: Corregido para usar la columna "U" ("Horas Netas Reclamo") en lugar de la "P" para el cálculo de horas.

- **Frontend:**
  - `web/static/panel.js`: El modal de correo consume el detalle de incidente para precargar asunto/cuerpo, usa `{{cantidad}}` en la plantilla y expone `openEmailModal(incidenteId)`; botón de descarga indica que guarda automáticamente.
  - `web/templates/panel.html`: Tooltip del botón de descarga EML aclara la persistencia automática.
  - `web/templates/panel.html`: Modales de conflicto específicos para cada escenario (upgrade, new strand).
  - `web/static/panel.js`: Lógica para manejar los nuevos flujos de resolución de conflictos y renderizar información adicional en tarjetas.
  - `web/static/panel.js`: Simplificación del modal de correo, con persistencia de destinatarios en `localStorage`.

- La corrección del SLA es crítica para la veracidad de los datos. Los informes generados antes de este cambio deben considerarse incorrectos en su cálculo de horas.
